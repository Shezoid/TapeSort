# TapeSort

### AbstractBuilderhandlersFactory
Интерфейс для фабрик, создающих цепочку обязанностей, обрабатывающих хеш таблицу в которую в качестве ключа записано название .env переменной, а в качестве значения соответсвенно его значение.
### AbstractTape
Интерфейс для работы с типом данных Tape, представлен в виде шаблона для любых целочисленных типов.
### AbstractTapeFactory
Интерфейс для всех абстрактных фабрик, создающих одну из реализаций AbstractTape.
### Config
Структура для хранения данных из .env файла. Предоставляет билдер для удобного создания.
### AbstractHandler
Интерфейс для хендлеров обрабатывающих хеш таблицу, о которой упоминалось выше и передают значения через методы билдера (Config::Builder)
### BaseHandler
Базовый класс для хендлеров, реализует только общий метод линковки хендлеров
### ConfigHandler<T>
Шаблонная реализация данных хендлеров, принимет в себя std::function указывающий на метод билдера для применения значения из хеш таблицы
### ConfigBuilderHandlerFactory
Фабрика создающая сразу цепочку для базовой обработки данных из .env
### EnvParser
Класс принимает путь к .env файлу и парсит его в хеш таблицу
### FileTape
Реализация ленты на основе файла
NB: для иничиализации его нужен путь к файлу (файл будет открыт в бинарном режиме), если хотите передать в него файл где числа записаны через пробел, то следует сначала выдать ему путь до пустого файла, далее воспользоваться его методом convertFromStream(std::istream&). После этого он считает данный поток и перепишет его в бинарном виде в файле, который был ему передан после этого он вернет ленту в начальное положение.
### InMemoryTapeAdapter
Данный класс представляет адаптер для вектора к интерфейсу ленты, был создан для более удобного дебаггинга и изолированного тестирования сортировщика.
### <...>TapeFactory
Классы создающие временные ленты, необходим для сортировщика, тк без него он не сможет создавать временные ленты (может, но будет нарушать SOLID).
### TapeSorter
Класс предназначен для сортировки входной ленты,
идея сортровки схожа с TimSort,
и заключается в том,
что мы последовательно разбиваем нашу входную ленту на массивы данных,
помещающаяеся в выделенной нам памяти.
Внутри памяти мы ее сортируем любой устойчивой сортировкой.
Далее мы вместо того,
чтобы с помощью стека выбирали какие две ленты на оптимальнее смерджить,
мы просто с помощью приорететной очереди (в ней учитывается значение элемента и какая по номеру это кусок
,дабы сохранить устойчивость сортировки)
определяем в какой из лент
на текущей позиции находится мин элемент, и просто достаем его
и записываем в выходную ленту его, после чего двигаем дальше выходную ленту
и ленту из которой его записали, обнавляя при этом свою позицию в 
очереди (эта лптимизация в разы эффективнее работает на ленте, 
за счет снижения накладных расходов на работу с самой лентой)

#### deprеcated
Была попытка написать мердж сорт на самих лентах для максимальной
оптимизации памяти, но не придумал эффективного алгоритма слияния
из-за того, что требуется каждый раз передвигать "магнитную головку" на
начало/конец ленты. + придется создавать огромное кол-во временных лент
